# 在线考试系统数据库设计（第三范式方案）

## 1. 设计目标

本文档用于记录在线考试系统在满足第三范式（3NF）目标下的数据库设计方案。该方案不直接修改当前系统，而是作为论文撰写、后续数据库重构与系统演进的依据。

设计目标如下：

1. 消除非原子字段，避免在单列中保存集合型数据。
2. 消除部分依赖与传递依赖，减少主数据冗余。
3. 明确主数据与业务数据边界，提高结构清晰度。
4. 通过外键和唯一约束提高数据一致性。
5. 为学生注册、考试安排、题库管理、成绩管理等核心业务提供规范的数据支撑。

---

## 2. 第三范式设计原则说明

第三范式要求：

1. 数据表先满足第一范式：字段不可再分，单元格内只保存一个值。
2. 再满足第二范式：非主属性完全依赖于主键，不能只依赖联合主键的一部分。
3. 最终满足第三范式：非主属性不能依赖其他非主属性，即不能存在传递依赖。

在本系统中，以下问题最典型：

1. `exam_arrangement.target_classes` 使用逗号分隔存储多个班级 ID，违反第一范式。
2. `sys_class` 同时保存 `department` 和 `major`，存在 `class -> major -> department` 的传递依赖。
3. `question_bank.subject` 与 `test_paper.subject` 均为文本字段，学科主数据未独立建模。
4. 学生、教师共用一张用户表，基础账号信息与角色扩展信息未拆分。

因此，采用“主数据独立建模 + 业务关系拆表”的方式更适合满足第三范式。

---

## 3. 数据库总体结构

重构后的数据库可划分为以下模块：

1. 组织架构模块：院系、专业、班级。
2. 用户模块：系统账号、学生档案、教师档案。
3. 题库与试卷模块：学科、题目、试卷、试卷题目关系。
4. 考试安排模块：考试安排、考试目标班级。
5. 作答与成绩模块：考试记录、答题明细、错题本。
6. 公共业务模块：公告等。

---

## 4. 规范化后的表结构设计

### 4.1 院系表 `department`

作用：保存学校院系主数据。

| 字段名 | 类型 | 说明 |
|---|---|---|
| department_id | int PK | 院系主键 |
| department_name | varchar(100) | 院系名称，唯一 |
| status | tinyint | 状态，1启用，0停用 |
| create_time | datetime | 创建时间 |

设计说明：

1. 院系名称不再重复分散到班级表、用户表、学科表中。
2. 统一由 `department` 维护院系主数据。

### 4.2 专业表 `major`

作用：保存专业主数据。

| 字段名 | 类型 | 说明 |
|---|---|---|
| major_id | int PK | 专业主键 |
| major_name | varchar(100) | 专业名称 |
| department_id | int FK | 所属院系 |
| status | tinyint | 状态 |
| create_time | datetime | 创建时间 |

设计说明：

1. 一个院系可以对应多个专业。
2. 专业通过 `department_id` 与院系关联。
3. 同一院系内专业名唯一。

### 4.3 班级表 `class_info`

作用：保存班级主数据。

| 字段名 | 类型 | 说明 |
|---|---|---|
| class_id | int PK | 班级主键 |
| class_name | varchar(100) | 班级名称 |
| major_id | int FK | 所属专业 |
| grade_year | int | 入学年级，如 2023 |
| status | tinyint | 状态 |
| create_time | datetime | 创建时间 |

设计说明：

1. 原有 `sys_class` 中的 `department`、`major` 被拆分出去。
2. 班级只依赖自身及专业主键，不再对院系存在传递依赖。
3. `grade_year` 独立建模，可直接支持“2020级到2026级”等业务场景。

### 4.4 系统用户表 `sys_user`

作用：保存全体用户的统一账号信息。

| 字段名 | 类型 | 说明 |
|---|---|---|
| user_id | int PK | 用户主键 |
| username | varchar(50) | 登录账号，唯一 |
| password | varchar(100) | 加密密码 |
| real_name | varchar(50) | 真实姓名 |
| role | varchar(20) | 角色：admin/teacher/student |
| phone | varchar(20) | 联系方式 |
| status | tinyint | 状态，1启用，0禁用 |
| create_time | datetime | 创建时间 |

设计说明：

1. `sys_user` 仅保存通用账号属性。
2. 与角色相关的专属字段不直接放入本表。

### 4.5 学生档案表 `student_profile`

作用：保存学生特有信息。

| 字段名 | 类型 | 说明 |
|---|---|---|
| user_id | int PK/FK | 对应系统用户 |
| class_id | int FK | 所属班级 |
| student_no | varchar(50) | 学号，唯一 |

设计说明：

1. 将学生与班级的关系从用户主表中拆出。
2. 注册时学生通过学号与班级信息写入本表。

### 4.6 教师档案表 `teacher_profile`

作用：保存教师特有信息，并明确教师负责学科，以支持教师仅管理对应学科题库、仅批阅对应学科试卷的业务规则。

| 字段名 | 类型 | 说明 |
|---|---|---|
| user_id | int PK/FK | 对应系统用户 |
| department_id | int FK | 所属院系 |
| teacher_no | varchar(50) | 工号，唯一 |
| subject_id | int FK | 负责学科 |

设计说明：

1. 教师与学生分开建模，便于后续扩展。
2. 教师不再依赖班级字段表达所属单位。
3. 通过 `subject_id` 明确教师负责学科，可作为题库权限控制与试卷批阅权限控制的依据。

### 4.7 学科表 `subject`

作用：保存题库和试卷所属学科。

| 字段名 | 类型 | 说明 |
|---|---|---|
| subject_id | int PK | 学科主键 |
| subject_name | varchar(100) | 学科名称 |
| department_id | int FK | 所属院系 |
| status | tinyint | 状态 |
| create_time | datetime | 创建时间 |

设计说明：

1. 原先散落在题库表、试卷表中的学科文本被提取为主数据。
2. 学科与院系建立明确归属关系。

### 4.8 题库表 `question_bank`

作用：保存试题。

| 字段名 | 类型 | 说明 |
|---|---|---|
| question_id | int PK | 题目主键 |
| subject_id | int FK | 所属学科 |
| type | varchar(20) | 题型 |
| title | text | 题干 |
| options | json | 选项 |
| answer | text | 标准答案 |
| difficulty | tinyint | 难度 |
| create_by | int FK | 创建教师 |
| create_time | datetime | 创建时间 |

设计说明：

1. `subject` 改为 `subject_id`，消除自由文本冗余。
2. `options` 目前仍保留为 JSON，属于工程折中方案。若要求更严格，可继续拆成 `question_option` 表。

### 4.9 试卷表 `test_paper`

作用：保存试卷模板。

| 字段名 | 类型 | 说明 |
|---|---|---|
| paper_id | int PK | 试卷主键 |
| paper_name | varchar(200) | 试卷名称 |
| subject_id | int FK | 所属学科 |
| total_score | int | 总分 |
| pass_score | int | 及格线 |
| duration | int | 考试时长 |
| create_by | int FK | 创建教师 |
| create_time | datetime | 创建时间 |

设计说明：

1. `subject` 改为 `subject_id`。
2. 试卷统一由题目关系表描述题目组成。

### 4.10 试卷题目关系表 `paper_question_rel`

作用：维护试卷与题目的多对多关系。

| 字段名 | 类型 | 说明 |
|---|---|---|
| id | int PK | 主键 |
| paper_id | int FK | 试卷 ID |
| question_id | int FK | 题目 ID |
| score | int | 题目分值 |
| sort_order | int | 排序号 |

设计说明：

1. 一张试卷可包含多道题。
2. 同一题可被不同试卷复用。
3. 增加唯一约束 `(paper_id, question_id)`，防止重复关联。

### 4.11 考试安排表 `exam_arrangement`

作用：保存考试任务。

| 字段名 | 类型 | 说明 |
|---|---|---|
| exam_id | int PK | 考试主键 |
| title | varchar(200) | 考试标题 |
| paper_id | int FK | 关联试卷 |
| start_time | datetime | 开始时间 |
| end_time | datetime | 结束时间 |
| status | varchar(20) | 状态 |
| create_by | int FK | 发布教师 |
| create_time | datetime | 创建时间 |

设计说明：

1. 原先 `target_classes` 逗号字段被彻底移除。
2. 考试与目标班级关系拆到独立表中。

### 4.12 考试目标班级表 `exam_target_class`

作用：保存考试覆盖的班级范围。

| 字段名 | 类型 | 说明 |
|---|---|---|
| id | int PK | 主键 |
| exam_id | int FK | 考试 ID |
| class_id | int FK | 班级 ID |

设计说明：

1. 一场考试可对应多个班级。
2. 同一班级也可参加多场考试。
3. 该表替代原先不规范的逗号拼接字段，满足第一范式并支撑第三范式设计。

### 4.13 学生考试记录表 `student_exam_record`

作用：保存学生的考试结果。

| 字段名 | 类型 | 说明 |
|---|---|---|
| record_id | int PK | 记录主键 |
| exam_id | int FK | 考试 ID |
| student_id | int FK | 学生用户 ID |
| total_score | int | 总分 |
| objective_score | int | 客观题得分 |
| subjective_score | int | 主观题得分 |
| status | varchar(20) | 状态 |
| create_time | datetime | 创建时间 |

设计说明：

1. 增加唯一约束 `(exam_id, student_id)`。
2. 保证同一学生在同一考试只有一条记录。

### 4.14 学生答题明细表 `student_answer_detail`

作用：保存某次考试中每道题的作答情况。

| 字段名 | 类型 | 说明 |
|---|---|---|
| detail_id | int PK | 明细主键 |
| record_id | int FK | 考试记录 ID |
| question_id | int FK | 题目 ID |
| answer_content | text | 学生答案 |
| is_correct | tinyint | 是否正确 |
| score | int | 得分 |

设计说明：

1. 增加唯一约束 `(record_id, question_id)`，防止同题重复写入。

### 4.15 错题本表 `wrong_question_book`

作用：记录学生的错题。

| 字段名 | 类型 | 说明 |
|---|---|---|
| id | int PK | 主键 |
| student_id | int FK | 学生 ID |
| question_id | int FK | 题目 ID |
| source_exam_id | int FK | 来源考试 |
| error_count | int | 错误次数 |
| last_error_time | datetime | 最后错误时间 |

设计说明：

1. 增加唯一约束 `(student_id, question_id)`。
2. 防止错题本中同一道题出现多条记录。

### 4.16 公告表 `sys_announcement`

作用：保存系统公告。

| 字段名 | 类型 | 说明 |
|---|---|---|
| id | int PK | 主键 |
| title | varchar(200) | 公告标题 |
| content | text | 公告内容 |
| is_top | tinyint | 是否置顶 |
| create_by | int | 发布人 |
| create_time | datetime | 发布时间 |

说明：

1. 本表本身不存在明显的传递依赖。
2. 保留原结构即可，已经基本满足第三范式要求。

## 5. 规范化前后核心差异

### 5.1 班级结构重构

原设计：

```text
sys_class(class_id, class_name, major, department)
```

问题：

1. 一个班级所属专业。
2. 一个专业又所属院系。
3. 因此 `department` 依赖于 `major`，而不是直接依赖于 `class_id`。
4. 形成传递依赖，不满足第三范式。

新设计：

```text
department -> major -> class_info
```

效果：

1. 院系、专业、班级各自独立。
2. 修改某个专业名称或院系名称时，不会影响大量班级记录。

### 5.2 考试目标班级拆表

原设计：

```text
exam_arrangement.target_classes = '1,2,3'
```

问题：

1. 非原子字段。
2. 难以查询、统计、扩展。

新设计：

```text
exam_arrangement
exam_target_class
```

效果：

1. 满足第一范式。
2. 为第三范式打下基础。
3. 支持考试与班级的规范多对多关系。

### 5.3 学科主数据独立化

原设计中：

1. 题库表有 `subject`。
2. 试卷表也有 `subject`。
3. 同一学科名称会在多表重复出现。

新设计中：

1. 学科独立为 `subject` 表。
2. 题目和试卷统一引用 `subject_id`。

效果：

1. 减少重复数据。
2. 便于学科统一维护。

## 6. 关键约束设计

为保证数据质量，设计中建议增加以下约束：

1. `sys_user.username` 唯一。
2. `student_profile.student_no` 唯一。
3. `teacher_profile.teacher_no` 唯一。
4. `paper_question_rel(paper_id, question_id)` 唯一。
5. `student_exam_record(exam_id, student_id)` 唯一。
6. `student_answer_detail(record_id, question_id)` 唯一。
7. `wrong_question_book(student_id, question_id)` 唯一。

这些约束能有效减少重复记录和业务异常。

---

## 7. 规范化后的优点

采用第三范式方案后，系统数据库将具有以下优点：

1. 结构更清晰，便于论文中的模块化描述。
2. 主数据维护集中，避免院系、专业、学科等字段在多处冗余。
3. 更容易扩展，例如新增学科、跨院系考试、教师归属调整等。
4. 查询逻辑更规范，便于后续编写接口与统计报表。
5. 数据一致性更好，减少脏数据、孤儿数据与重复数据。

---

## 8. 结论

当前在线考试系统若要严格满足第三范式，需要重点完成以下重构：

1. 将院系、专业、班级拆分为独立主数据表。
2. 将学科抽象为独立主数据表。
3. 将考试目标班级从逗号字段拆为关系表。
4. 将学生、教师角色扩展信息从用户主表中分离。
5. 为答题、错题、试卷题目关系补充唯一约束。

从论文撰写角度看，该方案能够较好体现数据库规范化设计思想，并说明系统在可维护性、可扩展性和数据一致性方面的提升。

## 9. E-R 关系说明

从概念模型角度分析，在线考试系统的核心实体包括院系、专业、班级、用户、学生档案、教师档案、学科、题目、试卷、考试安排、考试记录、答题明细和错题记录等。各实体之间的主要关系如下：

1. 一个院系可以包含多个专业，而一个专业只能隶属于一个院系。
2. 一个专业可以包含多个班级，而一个班级只能隶属于一个专业。
3. 一个系统用户若为学生，则对应唯一一条学生档案；学生档案只能归属一个班级。
4. 一个系统用户若为教师，则对应唯一一条教师档案；教师档案归属一个院系，并对应一个负责学科。
5. 一个院系可以开设多个学科，而一个学科仅归属于一个院系。
6. 一个学科可以包含多道题目，也可以关联多张试卷。
7. 一张试卷可以关联多道题目，一道题目也可以被多张试卷复用，因此试卷与题目之间构成多对多关系，并通过 `paper_question_rel` 实现。
8. 一场考试可以面向多个班级发布，一个班级也可以参加多场考试，因此考试与班级之间构成多对多关系，并通过 `exam_target_class` 实现。
9. 一个学生可以参加多场考试，而一场考试可以对应多个学生，因此学生与考试之间通过 `student_exam_record` 建立联系。
10. 一条考试记录可以对应多条答题明细，每条答题明细只对应一道题目。
11. 一个学生可以产生多条错题记录，但同一学生对同一道题在错题本中仅保留一条记录。

可将其简化表示为：

```text
院系 1 --- n 专业 1 --- n 班级
院系 1 --- n 学科
用户 1 --- 1 学生档案 n --- 1 班级
用户 1 --- 1 教师档案 n --- 1 院系
教师档案 n --- 1 学科
学科 1 --- n 题目
学科 1 --- n 试卷
试卷 n --- n 题目（通过试卷题目关系表）
考试 1 --- n 目标班级关系 n --- 1 班级
考试 1 --- n 学生考试记录 n --- 1 学生
学生考试记录 1 --- n 答题明细
学生 1 --- n 错题记录 n --- 1 题目
```

## 10. 表关系文字描述

为便于论文撰写，本节从关系数据库设计角度对主要数据表之间的联系进行文字化描述。

### 10.1 组织架构关系

组织架构采用“院系 - 专业 - 班级”三级结构。院系表作为顶层主数据表，一个院系可对应多个专业；专业表通过外键关联院系表，一个专业下可继续对应多个班级；班级表通过外键关联专业表，并通过 `grade_year` 字段表示入学届别。该设计避免了在班级表中重复存储院系名称与专业名称，从而消除了传递依赖。

### 10.2 用户与角色扩展关系

系统用户表仅保存通用账号属性，如用户名、密码、真实姓名、角色、联系方式和账号状态等。学生与教师的扩展信息分别存储在 `student_profile` 与 `teacher_profile` 中。学生档案通过 `class_id` 与班级表关联，用于明确学生所属班级；教师档案通过 `department_id` 与院系表关联，并通过 `subject_id` 指定负责学科。该设计使基础账号信息与角色扩展信息分离，减少字段空置和语义混杂问题。

### 10.3 学科、题目与试卷关系

学科表作为题库与试卷的统一主数据来源。题库表通过 `subject_id` 关联学科表，试卷表也通过 `subject_id` 关联学科表，从而避免学科名称在多个业务表中重复存储。试卷与题目之间采用中间表 `paper_question_rel` 建立多对多关系，同时保存题目在该试卷中的分值与排序信息。

### 10.4 考试安排与班级关系

考试安排表仅保存考试任务本身的基础属性，包括考试标题、关联试卷、开始时间、结束时间、状态和创建人等。考试面向的班级范围不再以逗号分隔字符串存放，而是由 `exam_target_class` 表单独维护。这一设计既满足第一范式，也便于进行考试班级范围查询、统计与权限控制。

### 10.5 成绩与作答关系

学生考试记录表用于记录学生参加某场考试后的总体结果，包括总分、客观题得分、主观题得分及批阅状态等。学生答题明细表按题目维度记录每一道题的答案内容、是否正确和实际得分。二者构成一对多关系，即一条考试记录可对应多条答题明细。

### 10.6 错题本关系

错题本表以学生和题目作为核心业务维度，记录学生错题的来源考试、错误次数及最近错误时间。通过唯一约束 `(student_id, question_id)`，可以保证同一学生对同一道题仅保留一条错题记录，从而避免重复数据。

## 11. 原设计与第三范式设计对比

| 对比维度 | 原设计 | 第三范式设计 | 优化效果 |
|---|---|---|---|
| 院系、专业、班级 | `sys_class` 一表混存 `department`、`major`、`class_name` | 拆分为 `department`、`major`、`class_info` | 消除传递依赖，结构更清晰 |
| 用户角色扩展 | 学生、教师共用一张用户表并混用附加信息 | `sys_user` + `student_profile` + `teacher_profile` | 角色边界更清晰，便于扩展 |
| 教师负责范围 | 未显式记录教师负责学科 | `teacher_profile.subject_id` | 支持按学科管理题库和批阅试卷 |
| 学科字段 | `question_bank.subject`、`test_paper.subject` 文本重复 | 独立 `subject` 表并统一引用 `subject_id` | 避免冗余，便于统一维护 |
| 考试目标班级 | `target_classes` 使用逗号分隔字符串 | `exam_target_class` 独立关系表 | 满足 1NF，查询与统计更规范 |
| 试卷题目关系 | 有关系表，但缺少唯一约束 | 增加 `(paper_id, question_id)` 唯一约束 | 避免题目重复关联到同一试卷 |
| 学生考试记录 | 结构基本可用，但缺少唯一性约束 | 增加 `(exam_id, student_id)` 唯一约束 | 避免重复考试记录 |
| 学生答题明细 | 缺少唯一性约束 | 增加 `(record_id, question_id)` 唯一约束 | 避免重复答题明细 |
| 错题本 | 同题可能重复记录 | 增加 `(student_id, question_id)` 唯一约束 | 保证错题记录唯一 |
| 数据一致性保障 | 主要依赖业务代码控制 | 外键 + 唯一约束 + 主数据拆分 | 数据一致性显著增强 |

## 12. 论文语言风格下的总结说明

从数据库规范化角度看，原在线考试系统虽然能够支撑基本的业务运行，但在组织架构建模、学科主数据管理以及考试目标班级表达等方面仍存在较为明显的冗余与结构性缺陷，尚未达到第三范式要求。其主要表现为：班级表中同时保存院系与专业信息，造成传递依赖；考试安排表中以逗号分隔形式存储多个班级标识，违反第一范式；题库与试卷中重复保存学科文本，造成主数据冗余。

为解决上述问题，本文提出了一套面向第三范式的数据库重构方案。该方案通过将院系、专业、班级和学科等基础信息独立为主数据表，并将教师、学生的角色扩展信息从用户主表中拆分出来，使各类非主属性均能够直接依赖于主键而非其他非主属性，从而有效消除传递依赖。同时，通过将考试目标班级关系、试卷与题目关系、学生考试记录与答题明细关系进行规范化建模，系统在数据一致性、查询效率和后续扩展能力方面都能够得到明显提升。

值得强调的是，在教师角色建模中增加“负责学科”字段，不仅使教师业务边界更加清晰，而且为实现“教师仅可管理对应学科题库、仅可批阅对应学科试卷”的权限控制机制提供了直接的数据支撑。这种设计体现了数据库结构与业务规则的统一，有助于提高系统设计的合理性和可解释性。

综上所述，第三范式数据库设计不仅能够降低在线考试系统中的数据冗余和异常更新风险，而且能够为后续的权限管理、资源归属控制和统计分析提供良好的结构基础，因此具有较高的理论意义和工程实践价值。
